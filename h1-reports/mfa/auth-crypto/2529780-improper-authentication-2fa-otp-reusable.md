# Improper Authentication - 2FA OTP Reusable After Generation of New OTP

- **CVEs:** None

## Summary
HackerOne's 2FA implementation fails to invalidate previous OTP codes when new codes are generated by Microsoft Authenticator. OTP tokens remain valid indefinitely despite the app cycling through new codes every 30 seconds, allowing attackers to reuse old OTPs for authentication. This significantly reduces the entropy of the 2FA mechanism and enables brute-force attacks.

## Attack scenario
1. Attacker obtains victim's HackerOne credentials through credential theft, phishing, or data breach
2. Victim has 2FA enabled with Microsoft Authenticator app
3. Attacker initiates login with victim's credentials and is prompted for OTP
4. Attacker captures or observes a valid OTP code displayed in Authenticator (or derives from common 6-digit patterns)
5. Even after 30-90 seconds when new OTPs are generated, the old OTP remains accepted by HackerOne's authentication system
6. Attacker successfully authenticates and gains account access, compromising private programs and sensitive data

## Root cause
HackerOne's OTP validation logic does not implement server-side timestamp validation or OTP invalidation upon generation of subsequent codes. The system accepts any previously valid OTP without checking if a newer OTP has been issued, violating TOTP standards (RFC 6238) which specify time-window based acceptance.

## Attacker mindset
An attacker who has compromised credentials seeks to bypass 2FA protections by exploiting the lack of OTP expiration. The reusability of old codes dramatically increases the attack surface - instead of a narrow 30-second window, they have unlimited time to use captured OTPs. Combined with the common nature of 6-digit codes, this enables practical brute-force attacks against the 2FA layer.

## Defensive takeaways
- Implement strict server-side OTP validation with timestamp verification (accept only current and previous time window)
- Invalidate OTP tokens immediately upon successful use and reject reuse attempts
- Implement rate limiting on failed OTP attempts per user/IP address
- Track OTP counter state and reject codes that are too far in the past per RFC 6238 specifications
- Add alerts for multiple failed OTP attempts or unusual authentication patterns
- Enforce proper TOTP time-window skew (typically ±1 window, not unlimited)
- Log all OTP validation attempts for security monitoring
- Consider requiring hardware security keys as backup 2FA method alongside TOTP

## Variant hunting
Check if other authentication mechanisms (backup codes, SMS OTP) have similar invalidation issues
Test if OTPs remain valid across different sessions or after password changes
Investigate whether OTP reuse works across different authentication endpoints or APIs
Examine if rate limiting on OTP attempts exists and is effective
Check for timing attacks or information leakage in OTP validation responses
Test if OTP tokens issued before 2FA setup remain valid after re-enabling 2FA
Verify behavior when system clocks are skewed or time synchronization issues occur
