# Absence of Token Expiry Leads to Unauthorized Login Access via OTP Brute Force

## Metadata
- **Source:** HackerOne
- **Report:** 766578 | https://hackerone.com/reports/766578
- **Submitted:** 2019-12-31
- **Reporter:** yogesh_ojha
- **Program:** Affirm
- **Bounty:** Not specified in report
- **Severity:** critical
- **Vuln:** Broken Authentication, Missing Token Expiration, Brute Force Attack, Weak Rate Limiting
- **CVEs:** None
- **Category:** auth-crypto

## Summary
The Affirm mobile app's login flow generates a non-expiring token that allows an attacker to brute force a 4-digit OTP indefinitely. An attacker can repeatedly submit OTP guesses against the same token without any timeout, enabling account takeover with minimal effort.

## Attack scenario
1. Attacker identifies target phone number and calls /api/v3/login/phone/ endpoint with the victim's phone number
2. Server responds with a permanent login token (SOMETOKEN) and response_url
3. Attacker uses brute force tools (Burp Intruder) to submit all 10,000 possible 4-digit OTP combinations (0000-9999) against /api/v3/login/phone/SOMETOKEN
4. Since token never expires, attacker can take unlimited time and attempts without restrictions
5. When correct OTP is guessed, server returns 200 OK with Affirm-Client session cookie and user_id
6. Attacker uses stolen Affirm-Client session to access victim's account and personal information via /api/v2/users/ endpoint

## Root cause
The authentication token generated by /api/v3/login/phone/ lacks expiration time, allowing indefinite brute force attempts. Additionally, the 4-digit OTP (0000-9999) provides only 10,000 possibilities with no rate limiting or attempt throttling.

## Attacker mindset
An attacker with knowledge of a target's phone number can systematically enumerate all possible OTP values without time pressure or blocking mechanisms. The lack of token expiry removes the urgency to crack the PIN quickly, enabling methodical brute force attacks at any pace.

## Defensive takeaways
- Implement strict token expiration (5-15 minutes) for login URLs/tokens
- Enforce rate limiting on OTP submission attempts (max 3-5 attempts per token)
- Lock out tokens after failed OTP attempts threshold is exceeded
- Implement exponential backoff or CAPTCHA after repeated failed attempts
- Log and monitor suspicious brute force patterns on authentication endpoints
- Use longer OTP values (6+ digits) or alphanumeric codes instead of 4-digit PINs
- Implement account lockout mechanisms after multiple failed authentication attempts
- Add IP-based rate limiting to prevent rapid-fire request floods

## Variant hunting
Check other authentication endpoints for similar token expiry issues (/api/v3/login/email/, /api/v3/login/password/)
Verify if password reset tokens have proper expiration
Test if session tokens have refresh mechanisms or absolute expiration limits
Search for other endpoints accepting SOMETOKEN format without validation
Check if API rate limiting can be bypassed using different User-Agent or Device-ID headers
Test if token reuse is possible across multiple accounts or devices
Verify OTP length consistency across different authentication flows

## Notes
The researcher demonstrated a complete attack chain from initial reconnaissance to account compromise. The vulnerability is particularly severe because: (1) it requires only the victim's phone number, (2) no rate limiting exists, (3) token never expires, (4) OTP space is small (10,000 possibilities), and (5) successful OTP grants full session access. The expiration date in the response ('3019-12-31') appears to be a test/placeholder value, further indicating inadequate token lifecycle management.

### Remediation
Rate limiting could be one of the fundamental solutions by limiting the number of the wrong OTP a user can submit.
The fundamental problem here is not that OTP is possible for Bruteforce, but the lack of token expiry generated for login purpose.

Luckily, there is a better way for this. When the user enters the number, and a password login URL/login is generated on the endpoint https://hackerone.affirm-odin.com/api/v3/login/phone/, the URL could be set invalid after a few OTP limits.

Once this is set to expiry, then to make another consecutive request to the endpoint https://hackerone.affirm-odin.com/api/v3/login/phone/SOMETOKEN would be automatically invalid. This should be done without even checking OTP to prevent brute-forcing. The login token generation on the endpoint https://hackerone.affirm-odin.com/api/v3/login/phone/ should be limited.
