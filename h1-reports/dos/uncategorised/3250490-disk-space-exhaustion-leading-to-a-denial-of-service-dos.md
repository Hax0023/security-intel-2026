# Disk Space Exhaustion DoS via Unlimited Debug Logging in cURL

- **CVEs:** None
- **Category:** uncategorised

## Summary
The cURL debug logging functions (tool_debug_cb and dump) lack size limits when writing trace output to files, allowing attackers to exhaust disk space by triggering large data transfers with --trace or --trace-ascii options enabled. An attacker can cause indefinite log file growth by forcing cURL to download/upload massive amounts of data, disrupting other services on the system.

## Attack scenario
1. Attacker sets up a malicious HTTP server that sends chunked responses with unlimited data
2. Attacker tricks or configures a victim system running cURL with --trace or --trace-ascii logging enabled to connect to the malicious server
3. cURL initiates connection and begins receiving unbounded HTTP response data
4. Debug callbacks (tool_debug_cb) write every header, data chunk, and SSL detail to the trace log file without size checks
5. Log file grows rapidly, consuming available disk space on the system
6. System experiences disk exhaustion, causing service disruptions, application crashes, and denial of service to legitimate users

## Root cause
The tool_debug_cb and dump functions in cURL perform unbounded fwrite/fprintf operations to trace log files without implementing any cumulative size limits, rate limiting, or rotation mechanisms. The code writes complete HTTP headers, data payloads, and formatted hex/ASCII representations of all transferred data without checking if the log file has exceeded a maximum size threshold.

## Attacker mindset
An attacker seeks to disrupt cURL-dependent services by exhausting system resources. By exploiting the verbose logging feature and forcing large data transfers, the attacker can achieve denial of service with minimal effort—simply hosting an unlimited data stream. This is particularly effective against automated tools and scripts that use cURL with debugging enabled in production environments.

## Defensive takeaways
- Implement maximum log file size limits (e.g., cap trace output at 100MB per session or implement rotating logs)
- Add rate limiting or sampling to debug output to prevent excessive writes during high-volume data transfers
- Disable verbose logging (--trace, --trace-ascii) by default in production; require explicit opt-in
- Implement log rotation mechanisms with size-based or time-based thresholds
- Monitor disk usage and alert when log files exceed specified thresholds
- Add safeguards to prevent trace files from being written to critical partitions
- Use separate log partitions to isolate log growth from critical system partitions
- Document security implications of debug flags in user-facing documentation

## Variant hunting
Check for similar unbounded logging in other cURL callbacks (progress callbacks, header callbacks)
Audit other tools using libcurl with verbose logging enabled for identical vulnerabilities
Search for fwrite/fprintf operations in debugging functions without size validation
Investigate whether --dump-header flag has similar exhaustion potential
Test if compressed transfers (gzip, deflate) bypass size calculations in logging

Check if CURLOPT_VERBOSE applies to other file handles beyond trace output

# Description

The tool_debug_cb function can write large amounts of debug data to a log file if the --trace or --trace-ascii options are used with a large volume of data. If an attacker can cause cURL to download or upload a very large amount of data (e.g., via a very large HTTP response or an unlimited upload), the log file generated by this debug function can grow indefinitely. This can lead to disk space exhaustion on the system where cURL is running, which in turn can disrupt other services running on the same server.
