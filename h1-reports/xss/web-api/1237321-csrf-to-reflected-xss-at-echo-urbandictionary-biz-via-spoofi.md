## Summary
The echo.urbandictionary.biz host reflects POST request bodies in HTML responses when accessing .html file extensions, allowing attackers to inject arbitrary JavaScript code. By combining CSRF with content-type spoofing using text/plain encoding, an attacker can bypass XSS filters and execute malicious scripts in victim browsers.

## Attack scenario
1. Attacker crafts HTML page containing a hidden form that targets echo.urbandictionary.biz/xsxsxs.html
2. Form uses POST method with text/plain enctype to bypass standard content-type validation
3. Attacker embeds JavaScript payload in form field name using CRLF injection (&#13;&#10;) to inject HTML/script tags
4. Victim visits attacker's page and form auto-submits via JavaScript
5. Browser sends POST request with injected payload due to CSRF vulnerability
6. Server reflects POST body with HTML content-type, executing JavaScript in victim's browser context

## Root cause
The application reflects user-supplied POST request data in HTTP responses without proper sanitization or encoding when the requested resource has a .html extension, and does not validate/restrict content-type headers, allowing CRLF injection to break out of form encoding and inject raw HTML/script tags.

## Attacker mindset
Exploit the echo endpoint's blind reflection of POST data combined with CSRF to achieve arbitrary code execution. The use of text/plain encoding and CRLF sequences demonstrates sophisticated understanding of HTTP encoding bypass techniques and form submission mechanics.

## Defensive takeaways
- Implement strict output encoding/escaping for all reflected user input regardless of file extension
- Validate and enforce expected content-types; reject requests with mismatched content-type headers
- Implement Content Security Policy (CSP) headers to restrict script execution
- Use SameSite cookie attribute to mitigate CSRF attacks
- Implement CSRF tokens for state-changing requests
- Avoid creating echo/reflection endpoints unless absolutely necessary; if required, treat as high-risk
- Sanitize all user input server-side before reflection in responses
- Implement input validation to reject CRLF sequences and other injection patterns

## Variant hunting
Test other file extensions (.htm, .shtml, .xml) for similar reflection behavior
Attempt POST request reflection on non-existent paths with various content-types
Test GET parameter reflection combined with content-type manipulation
Try double encoding and alternative CRLF representations (\r\n, %0d%0a)
Test for stored XSS if POST data is cached or logged anywhere
Check for mutation-XSS payloads that survive existing filters
Test HEAD requests and other HTTP methods for reflection
