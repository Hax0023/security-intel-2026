# XSS vulnerable parameter in location hash via convertro integration

## Summary
A reflected XSS vulnerability exists in Slack's live.js when processing the cvo_sid1 parameter from the URL hash used by the Convertro integration. The parameter is passed unsanitized to Convertro code, which can be exploited by manipulating the typ parameter to inject malicious JavaScript. An attacker can craft a malicious URL containing XSS payload that executes in the victim's browser when visited.

## Attack scenario
1. Attacker identifies that live.js processes cvo_sid1 parameter from location hash without sanitization
2. Attacker discovers that by chaining cvo_sid1 with typ parameter, malformed JavaScript can be generated by Convertro
3. Attacker crafts payload using typ parameter with JavaScript code, bypassing semicolon restrictions using %3b URL encoding
4. Attacker generates phishing URL: https://slack.com/is#?cvo_sid1=111&;typ=55577]")%3balert(document.cookie)%3b//
5. Attacker sends link to victim via email, chat, or social media
6. Victim clicks link, XSS payload executes in browser context, stealing cookies and enabling account takeover

## Root cause
Insufficient input validation and sanitization of the cvo_sid1 parameter in live.js before passing it to third-party Convertro integration code. Reliance on third-party (Convertro) sanitization without additional client-side validation. URL encoding bypass of semicolon restrictions allowed parameter injection.

## Attacker mindset
Opportunistic vulnerability discovery leveraging third-party integrations as attack surface. Attacker recognized that while Convertro sanitized directly, the combination of unsanitized input parameters could bypass protections. Targeting cookies and account credentials through phishing link distribution to achieve account takeover.

## Defensive takeaways
- Always sanitize and validate user input on the client-side, do not rely solely on third-party sanitization
- Implement Content Security Policy (CSP) headers to prevent inline script execution
- Use DOM APIs safely (textContent instead of innerHTML) when processing user-controlled data
- Encode all special characters properly when passing data to external integrations
- Do not trust hash parameters as they are user-controlled and not protected by CORS
- Implement additional output encoding when generating dynamic JavaScript
- Restrict allowed characters in parameters (whitelist approach) rather than blacklist
- Regular security testing of third-party integrations and their parameter handling

## Variant hunting
Search for other parameters processed from location.hash in live.js or similar tracking scripts
Test other third-party integrations (analytics, advertising pixels) for similar unsanitized parameter handling
Check for similar patterns where user input flows to window.eval() or dynamically executed code
Investigate other Slack domain/subdomain combinations for identical vulnerable code
Test if similar XSS exists through other hash parameters or query parameters
Look for character encoding bypasses on other parameters (URL encoding, unicode, etc.)

## Notes
This is a classic DOM-based reflected XSS via third-party integration. The reporter demonstrated good understanding by identifying the parameter chain (cvo_sid1 + typ) and encoding bypass technique. The vulnerability severity increased by account takeover potential through cookie theft. URL hash-based parameters are often overlooked in security reviews but are fully user-controlled and visible in referrer logs.
