# Reflected XSS in Error Pages via Null Byte Injection

## Metadata
- **Source:** HackerOne
- **Report:** 216812 | https://hackerone.com/reports/216812
- **Submitted:** 2017-03-28
- **Reporter:** sinkmanu
- **Program:** Nextcloud/Owncloud
- **Bounty:** Not specified in report
- **Severity:** High
- **Vuln:** Reflected Cross-Site Scripting (XSS), HTML Injection, Path Traversal/Null Byte Injection, Information Disclosure
- **CVEs:** CVE-2017-0891
- **Category:** web-api

## Summary
A reflected XSS vulnerability exists in Nextcloud/Owncloud error pages where unsanitized user input from URL parameters (files and dir) is reflected in HTML output. An attacker can inject arbitrary HTML/JavaScript by crafting a malicious URL containing special characters like null bytes, bypassing some security controls. The vulnerability also discloses full server file paths.

## Attack scenario
1. Attacker crafts a malicious URL containing null byte injection in the 'files' parameter (e.g., files=%00) and injects JavaScript/HTML in the 'dir' parameter
2. Attacker sends the crafted link to a logged-in user via email, chat, or social engineering
3. Victim clicks the link while authenticated to Nextcloud/Owncloud
4. The application processes the request and reflects the unsanitized input in an error page HTML response
5. Victim's browser executes the injected JavaScript in the context of the Nextcloud domain
6. Attacker can steal session cookies, redirect to phishing page mimicking login, capture credentials, or perform actions on behalf of the user

## Root cause
User-controlled input from URL parameters ('files' and 'dir') is reflected directly into HTML error page responses without proper output encoding or sanitization. While HTTP security headers (CSP, X-Frame-Options) provide some mitigation, they are not universally supported across all browsers and are not defense-in-depth controls.

## Attacker mindset
Reconnaissance-focused: The attacker identified that security headers exist but recognized browser compatibility gaps as an attack surface. They understood that error pages are often overlooked during security reviews and that authenticated user targeting increases success probability. The null byte injection technique suggests knowledge of how different parsers handle special characters differently.

## Defensive takeaways
- Always sanitize and HTML-encode all user-supplied input before reflecting it in HTTP responses, regardless of context
- Implement whitelist-based input validation on 'files' and 'dir' parameters; reject null bytes and other special characters
- Apply output encoding appropriate to context (HTML encoding for HTML context, JavaScript encoding for JS, URL encoding for URLs)
- Do not rely solely on HTTP security headers as the only defense; implement defense-in-depth
- Implement Content Security Policy (CSP) with strict settings and test browser compatibility
- Use templating engines with auto-escaping features to prevent accidental unencoded output
- Conduct security code review of error handling and logging paths, as these are often overlooked
- Sanitize error messages to avoid information disclosure (full paths, system details)
- Implement automated XSS testing in CI/CD pipeline for all user input reflection points

## Variant hunting
Search for other download/file serving endpoints that may reflect parameters in error responses
Test all AJAX endpoints for reflected XSS vulnerabilities
Check error handlers in other applications (e.g., admin panels, API endpoints) for similar patterns
Test POST parameters in addition to GET parameters for the same vulnerability
Investigate other special characters (e.g., Unicode, double-encoding) that might bypass filters
Look for stored XSS variants where file paths containing injected payloads could be saved and later reflected
Test error pages generated by missing files, permission denials, and resource exhaustion scenarios
