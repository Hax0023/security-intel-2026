# Stored XSS in HR Training Report Generation - goal[1][Costs] Parameter

- **CVEs:** None
- **Category:** web-api

## Summary
A stored XSS vulnerability exists in the HR Training report generation endpoint (/HRO/Training/idpgenerate.php) where user-supplied input in the goal[1][Costs] parameter is not properly sanitized or encoded before being stored and displayed. An attacker can inject malicious JavaScript that executes when authorized personnel view the generated training report, enabling session hijacking and data theft.

## Attack scenario
1. Attacker accesses the HR Training form at the vulnerable endpoint with valid session credentials
2. Attacker injects JavaScript payload (e.g., <ScRiPt>alert(9639)</ScRiPt>) into the goal[1][Costs] form field via multipart POST request
3. Application stores the malicious payload in the database without sanitization or encoding
4. HR supervisor or manager requests to view the generated training report containing the injected goal data
5. Malicious script executes in the victim's browser context with their session privileges
6. Attacker steals session cookies, credentials, or sensitive HR data from the report view

## Root cause
The application fails to implement proper input validation and output encoding for array-based form parameters. The goal[1][Costs] field accepts arbitrary HTML/JavaScript content that is stored persistently and rendered without escaping when the report is generated. The use of mixed-case script tags (<ScRiPt>) suggests inadequate filter implementation.

## Attacker mindset
An internal user or external attacker with form access exploits the report system to target HR administrators and supervisors who have higher privileges. By injecting persistent XSS, the attacker gains indirect access to sensitive HR data (employee records, performance reviews, compensation) through administrative sessions.

## Defensive takeaways
- Implement strict input validation on all form parameters, including nested array fields, using whitelist approach for expected data types and formats
- Apply context-appropriate output encoding (HTML entity encoding for HTML context) to all user-controlled data rendered in reports
- Use templating engines with auto-escaping features (e.g., Twig, Blade) instead of manual echo statements
- Implement Content Security Policy (CSP) headers to prevent inline script execution
- Perform comprehensive XSS testing on all form endpoints, including edge cases with mixed-case tags and array parameters
- Use framework-provided sanitization functions rather than custom filters
- Implement automated security scanning in the development pipeline to detect XSS before deployment
- Apply principle of least privilege to HR report access and audit access logs

## Variant hunting
Test other nested array parameters (development[n][Costs], development[n][Activity]) for similar XSS
Attempt double-encoding and alternative payload encodings (unicode, hex) to bypass filters
Test other form endpoints in /HRO/Training/ for similar vulnerabilities
Check if similar vulnerable patterns exist in other employee/report-related modules
Test reflected XSS in query parameters used for report generation or filtering
Investigate whether other HTML tags (img, svg, iframe, event handlers) are properly filtered
Test for DOM-based XSS in client-side report rendering if JavaScript processes the data

- T1539 - Steal Web Session Cookie
- T1041 - Exfiltration Over C2 Channel

## Notes
The report appears to be from an HR/training system managing employee development and performance goals. The vulnerability's impact is elevated due to the HR context where reported data contains sensitive employee and compensation information. The payload uses case-variation obfuscation, indicating awareness of basic XSS filters. The use of multipart/form-data and the specific nested array parameter structure (goal[1][Costs]) suggests a custom form handling system without standard framework protections.
