# XSS in Private Messages

## Metadata
- **Source:** HackerOne
- **Report:** 293105 | https://hackerone.com/reports/293105

- **Program:** VKontakte (VK.com)
- **Bounty:** Not specified in report

- **Vuln:** Cross-Site Scripting (Stored XSS), Input Validation Bypass, Insufficient Output Encoding
- **CVEs:** None
- **Category:** web-api

## Summary
A stored XSS vulnerability exists in the private messaging functionality where user input is not properly filtered or encoded. An attacker can inject malicious JavaScript code into private messages that executes in the recipient's browser when they view the message.

## Attack scenario
1. Attacker identifies that the private message input field lacks proper input validation and output encoding
2. Attacker crafts a message containing malicious JavaScript payload (e.g., <script>alert('XSS')</script> or similar variant)
3. Attacker sends the crafted message to a victim via private messaging
4. Victim receives and opens the private message containing the stored XSS payload
5. The JavaScript code executes in the victim's browser context, potentially allowing session hijacking, credential theft, or account compromise
6. Researcher notes that additional usernames with special characters may still be exploitable due to incomplete patching

## Root cause
The application fails to implement proper input sanitization and output encoding in the private messaging feature. The developers previously patched a vulnerability allowing special characters in usernames via mobile version but left other bypass vectors unpatched, indicating incomplete security review.

## Attacker mindset
An attacker would recognize that messaging features are common targets for XSS due to their persistence and direct user interaction. The attacker notes that the vendor's partial fix suggests other vectors remain unexploited, making this an attractive target for privilege escalation or user account compromise at scale.

## Defensive takeaways
- Implement comprehensive input validation on all user-supplied data, not just specific fields
- Apply context-appropriate output encoding (HTML encoding for HTML context, JavaScript encoding for JS context)
- Use a security-focused templating engine or library that auto-escapes output by default
- Conduct thorough security reviews when patching similar vulnerabilities to identify and close related attack vectors
- Implement Content Security Policy (CSP) headers to mitigate XSS impact
- Add automated security testing for XSS patterns in all user input fields
- Consider using a Web Application Firewall (WAF) as a defense-in-depth measure

## Variant hunting
Search for similar XSS vulnerabilities in other communication features (comments, posts, group messages), username/profile fields with special character handling, and any recently patched XSS locations where similar bypass techniques may apply. Focus on mobile application endpoints which were mentioned as previously exploitable.
