## Summary
A CSRF vulnerability combined with XSS allows an attacker to inject malicious JavaScript through unvalidated form parameters. The application fails to sanitize user input in multiple form fields, enabling arbitrary code execution in victims' browsers through crafted POST requests.

## Attack scenario
1. Attacker identifies a POST endpoint accepting user input without CSRF token validation
2. Attacker crafts malicious payload containing XSS (e.g., "><img src=x onerror=prompt``>) in form parameters
3. Attacker creates a CSRF PoC HTML page that auto-submits the form when visited
4. Attacker tricks a logged-in user into visiting the malicious page (via email, social media, etc.)
5. User's browser automatically submits the POST request with malicious payload to vulnerable endpoint
6. Application stores or reflects the XSS payload, executing JavaScript in victim's session context

## Root cause
The application lacks two critical security controls: (1) CSRF token validation on state-changing POST requests, and (2) output encoding/input sanitization of user-supplied data before rendering in responses. Multiple form fields are vulnerable to injection attacks.

## Attacker mindset
An attacker recognizes that combining CSRF with XSS significantly amplifies impact - CSRF bypasses the need for user interaction beyond page visit, while XSS enables session hijacking, credential theft, and malware distribution. The presence of multiple vulnerable parameters suggests systemic input handling failures.

## Defensive takeaways
- Implement CSRF tokens (synchronizer tokens or SameSite cookies) on all state-changing requests
- Apply output encoding appropriate to context (HTML entity encoding, JavaScript escaping, URL encoding)
- Implement Content Security Policy (CSP) headers to restrict script execution
- Validate and sanitize all user inputs server-side using allowlists
- Use security headers: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection
- Implement HttpOnly and Secure flags on session cookies
- Perform code review focusing on all data flow paths from user input to output
- Use templating engines with auto-escaping capabilities

## Variant hunting
Search for similar patterns: other POST endpoints lacking CSRF tokens, form fields accepting unvalidated text parameters, instances where user input flows directly to response without sanitization, especially in educational/classroom management systems or any CMS accepting structured form data

## Impact

Malicious JavaScript has access to all the same objects as the rest of the web page, including access to cookies and local storage, which are often used to store session tokens. If an attacker can obtain a user's session cookie, they can then impersonate that user.

Furthermore, JavaScript can read and make arbitrary modifications to the contents of a page being displayed to a user. Therefore, XSS in conjunction with some clever social engineering opens up a lot of possibilities for an attacker.
