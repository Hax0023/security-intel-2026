# Login with Google Not Authenticated on iOS App - ID Token Validation Bypass

- **CVEs:** None

## Summary
The Instacart iOS app's Google login endpoint fails to properly validate ID tokens, accepting tokens generated for other applications. An attacker can use ID tokens from third-party apps (e.g., Meetup) to authenticate as arbitrary users without owning their Google accounts, leading to complete account takeover.

## Attack scenario
1. Attacker performs Google login through any third-party app (e.g., Meetup) to obtain a valid Google ID token
2. Attacker extracts the id_token from the third-party app's login response
3. Attacker crafts a request to https://www.instacart.com/api/v2/users/google_login_auth with the borrowed id_token
4. Backend fails to validate that the token's aud (audience) claim matches Instacart's client_id
5. API accepts the token and creates/logs in a session for the target user
6. Attacker gains unauthorized access to the victim's Instacart account without knowing their credentials

## Root cause
The backend endpoint does not perform proper ID token validation according to OAuth 2.0 specifications. Specifically, it fails to verify: (1) the token's 'aud' (audience) claim against the app's registered client_id, (2) the token's signature using Google's public keys, (3) the token's issuer claim, and (4) the token's expiration. This allows tokens issued to other applications to be reused.

## Attacker mindset
An attacker with knowledge of OAuth 2.0 vulnerabilities would recognize that the backend is accepting tokens from any Google app without audience validation. They would attempt to obtain tokens from publicly available apps, then reuse them to compromise high-value accounts. For e-commerce platforms like Instacart, this enables access to payment methods, addresses, and order history.

## Defensive takeaways
- Always validate the 'aud' (audience) claim in ID tokens against your application's client_id
- Verify ID token signatures using Google's publicly available certificates
- Validate the 'iss' (issuer) claim matches Google's expected issuer URL
- Check token expiration time and reject expired tokens
- Use Google's official libraries for token validation rather than custom implementations
- Implement server-side verification for all OAuth tokens, never trust client-provided tokens
- For sensitive operations, validate tokens at every API call, not just login
- Log and monitor suspicious authentication patterns (e.g., tokens from multiple apps)

## Variant hunting
Check if other OAuth providers (Facebook, Apple Sign-In) have similar validation bypasses
Test if access_token parameter also lacks proper validation
Verify if the vulnerability extends to password reset or account recovery endpoints
Check if other endpoints accepting id_token have the same validation gap
Test if tokens can be reused across different sessions or devices
Investigate if the client_id parameter can be manipulated or if validation is skipped when not provided

## Notes
This is a textbook OAuth 2.0 implementation failure with severe business impact. The vulnerability allows account takeover without any user interaction or compromise of the target's Google credentials. The report correctly references Google's official documentation on backend authentication. This vulnerability likely affects all users and platforms (web may have similar issues). Remediation requires immediate backend updates to implement proper ID token validation before issuing session tokens.

This means ​an app like Meetup can access the user accounts of your users using the id_token generated by Meetup. This can compromise your user's accounts.

The proper way of authenticating the user is mentioned here:
